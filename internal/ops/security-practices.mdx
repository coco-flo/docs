## Credential storage

**All credentials live in Bitwarden.** No exceptions. Never store credentials in:

- Mintlify docs (including this one)
- Linear issue descriptions
- Lovable prompts
- Slack or group chat messages
- Notes apps or email drafts

If a credential appears in any of the above, rotate it immediately.

| Clarely Admin - Root Account | Heimdall root email, password |

## Heimdall access

Heimdall at `heimdall.clargate.tools` is the admin control plane. Access rules:

- Heimdall URL is not publicly linked. Share only via Bitwarden secure share or direct trusted communication.
- Always use the `super_admin` role for daily operations. Root is the break-glass account.
- Root credentials are Authy-backed (TOTP app-based). Never share or document the TOTP seed outside Bitwarden.
- Every action in Heimdall is audit-logged. The log is append-only and hash-chained.

## Secret rotation schedule

| Secret | Rotate when | Process |
| --- | --- | --- |
| ADMIN\_BRIDGE\_SECRET | If exposed, or every 6 months | Update in both Supabase projects' secrets panels. Notify AppDev. Update Bitwarden. |
| Stripe webhook signing secret | If exposed | Rotate in Stripe dashboard. Update in Supabase env vars. Update Bitwarden. |
| Resend API key | If exposed | Rotate in Resend dashboard. Update in Supabase env vars. Update Bitwarden. |
| Heimdall root password | If exposed, or annually | Update in Bitwarden and Heimdall admin panel. |
| Supabase service role keys | If exposed only | Update in AppDev environment. Update Bitwarden. |

## If a secret is exposed

1. Rotate the secret immediately (don't investigate first - rotate first)
2. Update all systems that use the secret
3. Update Bitwarden
4. Review audit logs to determine what was accessed
5. If customer data was potentially accessed, assess scope before deciding on disclosure

A secret in a Lovable prompt is considered exposed. Rotate immediately.

## Supabase security notes

- The `anon` key is public by design (used in the browser). Row Level Security protects data at the database level.
- The `service_role` key bypasses RLS. It is used only in Edge Functions, never in frontend code.
- Never commit the service role key to any repository or include it in Lovable prompts.

## Backup posture

Heimdall runs an hourly backup of clarely.app customer data to a private Supabase Storage bucket. Verify:

- Hourly runs are completing successfully (check Heimdall Backup overview)
- Daily runs are completing (check History tab)
- Restore has been tested at least once (quarterly)
